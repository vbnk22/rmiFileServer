2. Cel i zakres projektu

Cel: Stworzenie rozproszonej aplikacji serwera plików umożliwiającej wymianę plików między wieloma klientami za pośrednictwem protokołu Java RMI. System zapewnia kontrolę dostępu opartą na rolach (administrator / użytkownik) oraz bezpieczną współbieżną obsługę wielu klientów jednocześnie.

Przyjęte założenia

  • Komunikacja odbywa się wyłącznie przez Java RMI (rejestr na porcie 5555).
  • Dane użytkowników i metadane plików przechowywane są w pamięci operacyjnej (in-memory); pliki fizyczne zapisywane są na dysku serwera w katalogu storage/files/.
  • System ról: ADMIN (pełen dostęp) i USER (upload/download).
  • Uwierzytelnianie oparte na sesjach (UUID token).
  • Wiele klientów może jednocześnie pobierać ten sam plik (odczyt współbieżny).
  • Operacje zapisu i usuwania plików są wyłączne (blokada zapisu).

Zakres funkcjonalny

  • Rejestracja i usuwanie użytkowników przez administratora.
  • Logowanie z przydzielaniem tokenu sesji (UUID).
  • Wylistowanie plików dostępnych na serwerze.
  • Wysyłanie pliku na serwer (upload) — klient przesyła bajty przez RMI.
  • Pobieranie pliku z serwera (download) — serwer zwraca bajty przez RMI.
  • Usuwanie pliku przez administratora z potwierdzeniem.

5. Model współbieżności i synchronizacji

Identyfikacja i rozwiązanie sekcji krytycznych

1. Repozytorium plików — FileRepositoryImpl

Problem: Wielu klientów może jednocześnie pobierać, wgrywać lub usuwać pliki. Bez synchronizacji możliwe są:

  • Race condition przy zapisie — dwa równoległe uploady mogą nadpisać dane na dysku.
  • TOCTOU (Time-Of-Check-Time-Of-Use) — klient sprawdza, czy plik istnieje (findFileByName), a w oknie między sprawdzeniem a odczytem (downloadFile) admin usuwa ten plik.

Rozwiązanie: ReentrantReadWriteLock

  • Blokada odczytu (read lock) — downloadFile(), listFiles(), findFileByName(): wiele wątków jednocześnie może trzymać blokadę odczytu.
  • Blokada zapisu (write lock) — uploadFile(), deleteFile(): blokada wyłączna.
  • Metoda findAndDownload() — atomowe wyszukanie i odczyt pod jedną blokadą odczytu, eliminuje TOCTOU.
// Atomowe find+read — zapobiega TOCTOU race condition:
public byte[] findAndDownload(String fileName) {
    rwLock.readLock().lock();
    try {
        FileMetadata meta = files.stream()
                .filter(f -> f.getFileName().equals(fileName))
                .findFirst()
                .orElseThrow(() -> new RuntimeException("File not found: " + fileName));
        return Files.readAllBytes(storageDir.resolve(meta.getFileStoredName()));
    } finally {
        rwLock.readLock().unlock();
    }
}

// uploadFile — wyłączny zapis + zwolnienie pamięci:
public void uploadFile(FileMetadata meta) {
    rwLock.writeLock().lock();
    try {
        Files.write(storageDir.resolve(meta.getFileStoredName()), meta.getFileData());
        meta.setFileData(null);  // bajty zwolnione — brak wycieku pamięci
        files.add(meta);
    } finally {
        rwLock.writeLock().unlock();
    }
}

2. Lista użytkowników — UserRepositoryImpl

Problem: Jednoczesna rejestracja i usuwanie użytkowników z wielu klientów może prowadzić do ConcurrentModificationException.

Rozwiązanie: CopyOnWriteArrayList — operacje odczytu bezpieczne bez blokady. Każda operacja zapisu tworzy kopię listy, co gwarantuje atomowość.

3. Sesje użytkowników — AuthServiceImpl

Rozwiązanie: Dwie mapy ConcurrentHashMap (sesja→login, sesja→rola). Wszystkie operacje atomowe, bez potrzeby jawnej synchronizacji.

Brak zakleszczeń (Deadlock-free)

System nie jest podatny na zakleszczenia, ponieważ:

  • Każdy zasób ma jeden lock (rwLock w FileRepositoryImpl, ConcurrentHashMap w AuthServiceImpl) — brak możliwości cyklicznego oczekiwania.
  • Blokady zawsze zwalniane w bloku finally — gwarantowana inwolucja nawet przy wyjątku.
  • Brak zagnieżdżonego pobierania wielu blokad w jednym wątku.

6. Zarządzanie pamięcią i wydajność

Przechowywanie danych

DaneGdzieTrwałość
Metadane plikówIn-memory (CopyOnWriteArrayList<FileMetadata>)Utracone po restarcie serwera
Pliki fizyczneDysk: storage/files/{UUID}Trwałe na dysku
Konta użytkownikówIn-memory (CopyOnWriteArrayList<UserDTO>)Utracone po restarcie; admin odtwarzany automatycznie
SesjeIn-memory (ConcurrentHashMap)Utracone po restarcie

Zarządzanie pamięcią — kluczowe decyzje

  • Zwolnienie bajtów po uploadzie: Po zapisaniu pliku na dysk wywołujemy meta.setFileData(null). Bez tego każdy wgrany plik trzymałby bajty w heap przez cały czas działania serwera.
  • listFiles() korzysta z fileSize: Pole fileSize w FileMetadata ustawiane raz przy tworzeniu, dzięki czemu mapToDTO() nie musi odczytywać fileData (które jest już null).
  • Bajty w RMI: Podczas uploadu/downloadu byte[] jest trzymany w pamięci JVM tylko przez czas trwania wywołania RMI — serializacja/deserializacja zarządzana przez JVM.

Ograniczenia

  • Brak trwałości metadanych — restart serwera wymaga ponownego wgrania plików (fizyczne pliki pozostają na dysku).
  • Brak TTL dla sesji — zalogowane sesje nie wygasają automatycznie.
  • Brak limitu rozmiaru pliku — bardzo duże pliki mogą obciążyć heap podczas transferu RMI.
  • Jeden ReadWriteLock na cały zbiór plików — przy bardzo dużej liczbie klientów piszących jednocześnie może stać się wąskim gardłem (rozwiązanie: lock per-plik).

7. Instrukcja wdrożenia i obsługi

Wymagania

  • Java JDK 19 lub nowszy
  • Apache Maven 3.6+
  • Obie strony (klient i serwer) muszą mieć dostęp do tego samego modułu common

Krok 1 — Kompilacja

mvn compile

Kompiluje wszystkie trzy moduły (common, server, client).

Krok 2 — Uruchomienie serwera

cd server
java -cp target/classes;../common/target/classes ServerMain

Serwer uruchamia RMI Registry na porcie 5555 i rejestruje trzy usługi. Automatycznie tworzone jest konto admina: login: admin, hasło: admin.

8. Ograniczenia i testy poprawności

Maksymalna liczba jednoczesnych klientów

Java RMI tworzy wątek per wywołanie przy użyciu puli wątków JVM. Domyślnie brak twardego limitu — liczba jednoczesnych klientów ograniczona jest zasobami systemowymi. System działa stabilnie przy 10–20 jednoczesnych klientach w środowisku laboratoryjnym.

Test 1 — równoczesne pobieranie pliku (10 wątków)

Scenariusz: 10 wątków klienckich jednocześnie pobiera ten sam plik z serwera.

ExecutorService pool = Executors.newFixedThreadPool(10);
CountDownLatch latch = new CountDownLatch(1);
List<byte[]> results = new CopyOnWriteArrayList<>();

for (int i = 0; i < 10; i++) {
    pool.submit(() -> {
        try {
            latch.await(); // wszystkie wątki startują jednocześnie
            byte[] data = fileService.downloadFile("test.txt");
            results.add(data);
        } catch (Exception e) { e.printStackTrace(); }
    });
}
latch.countDown();
pool.shutdown();
pool.awaitTermination(30, TimeUnit.SECONDS);

assert results.size() == 10;
assert results.stream().allMatch(d -> d != null && d.length > 0);

Oczekiwany wynik: Wszystkie 10 wątków otrzymuje identyczne, kompletne bajty pliku. Brak wyjątków.

Dlaczego działa: readLock() pozwala wielu wątkom jednocześnie trzymać blokadę odczytu — żaden nie blokuje pozostałych.

Test 2 — równoczesny upload i download (TOCTOU)

Scenariusz: Wątek A pobiera plik, wątek B usuwa ten sam plik jednocześnie.

Bez ochrony (błędny kod): Możliwy scenariusz — A sprawdza findFileByName (plik istnieje), B wywołuje deleteFile (plik usunięty), A wywołuje downloadFile (RuntimeException — plik nie istnieje na dysku).

Z ochroną (findAndDownload): Operacja find+read odbywa się atomowo pod jedną blokadą odczytu. deleteFile czeka na blokadę zapisu — nie może wejść w sekcję krytyczną, dopóki download się nie zakończy. Wynik: albo A pobierze plik w całości, albo B usunie go i A dostanie czytelny błąd "File not found".

Test 3 — kolejkowanie zapisów

Scenariusz: 5 wątków wgrywa różne pliki jednocześnie.

Wynik: Wątki ustawiają się w kolejce po writeLock. Każdy plik zapisywany atomowo (meta + bajty). Po zakończeniu wszystkich uploadów lista plików zawiera dokładnie 5 pozycji.
