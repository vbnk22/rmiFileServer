2. Cel i zakres projektu

Cel: Stworzenie rozproszonej aplikacji serwera plików umożliwiającej wymianę plików między wieloma klientami za pośrednictwem protokołu Java RMI. System zapewnia kontrolę dostępu opartą na rolach (administrator / użytkownik) oraz bezpieczną współbieżną obsługę wielu klientów jednocześnie.

Przyjęte założenia

  • Komunikacja odbywa się wyłącznie przez Java RMI (rejestr na porcie 5555).
  • Metadane plików i konta użytkowników są persystowane w plikach CSV (storage/files_metadata.csv, storage/users.csv) — dane przeżywają restart serwera.
  • Pliki fizyczne zapisywane są na dysku serwera w katalogu storage/files/ pod nazwą UUID.
  • System ról: ADMIN (pełen dostęp) i USER (upload/download/wyszukiwanie).
  • Uwierzytelnianie oparte na sesjach (UUID token).
  • Wiele klientów może jednocześnie pobierać różne pliki — blokowanie drobnoziarniste per-plik.
  • Operacje zapisu i usuwania konkretnego pliku są wyłączne dla tego pliku, nie blokując operacji na innych plikach.
  • Rozmiar wgrywanego pliku ograniczony do 50 MB.

Zakres funkcjonalny

  • Rejestracja i usuwanie użytkowników przez administratora (persystowane w CSV).
  • Logowanie z przydzielaniem tokenu sesji (UUID).
  • Wylistowanie plików dostępnych na serwerze.
  • Wyszukiwanie pliku po nazwie — zwraca metadane (nazwa, rozmiar, UUID na dysku).
  • Wysyłanie pliku na serwer (upload) — limit 50 MB, klient przesyła bajty przez RMI.
  • Pobieranie pliku z serwera (download) — serwer zwraca bajty przez RMI.
  • Usuwanie pliku przez administratora z potwierdzeniem.

5. Model współbieżności i synchronizacji

Identyfikacja i rozwiązanie sekcji krytycznych

1. Repozytorium plików — FileRepositoryImpl

Problem: Wielu klientów może jednocześnie pobierać, wgrywać lub usuwać pliki. Bez synchronizacji możliwe są:

  • Race condition przy zapisie — dwa równoległe uploady tego samego pliku mogą nadpisać dane na dysku.
  • TOCTOU (Time-Of-Check-Time-Of-Use) — klient sprawdza, czy plik istnieje, a w oknie między sprawdzeniem a odczytem admin usuwa ten plik.
  • Race condition przy zapisie CSV — dwa równoległe uploady różnych plików mogą jednocześnie pisać do files_metadata.csv i wzajemnie nadpisać zmiany.

Rozwiązanie: blokowanie drobnoziarniste (per-file lock)

Zamiast jednego globalnego locka na wszystkie pliki zastosowano mapę locków: ConcurrentHashMap<String, ReentrantReadWriteLock> indeksowaną nazwą pliku. Dzięki temu klienci operujący na różnych plikach nie blokują się wzajemnie — kolejkowanie dotyczy wyłącznie równoległego dostępu do tego samego pliku.

  • Read lock per-plik — downloadFile(), findFileByName(): wiele wątków jednocześnie może pobierać różne (lub ten sam) pliki.
  • Write lock per-plik — uploadFile(), deleteFile(): wyłączny dostęp do konkretnego pliku, inne pliki dostępne równolegle.
  • Metoda findAndDownload() — atomowe wyszukanie i odczyt pod jedną blokadą odczytu per-plik, eliminuje TOCTOU.
  • csvLock (ReentrantLock) — osobna blokada wyłącznie do zapisu CSV; trzymana tylko przez czas zapisu pliku metadanych, nie przez cały czas I/O dysku.
// Mapa locków — jeden ReadWriteLock na każdy plik (klucz = oryginalna nazwa)
private final ConcurrentHashMap<String, ReentrantReadWriteLock> fileLocks = new ConcurrentHashMap<>();
private final ReentrantLock csvLock = new ReentrantLock();

private ReentrantReadWriteLock lockFor(String fileName) {
    return fileLocks.computeIfAbsent(fileName, k -> new ReentrantReadWriteLock());
}

// Atomowe find+read — zapobiega TOCTOU race condition:
public byte[] findAndDownload(String fileName) {
    ReentrantReadWriteLock lock = fileLocks.get(fileName);
    if (lock == null) throw new RuntimeException("File not found: " + fileName);
    lock.readLock().lock();
    try {
        FileMetadata meta = files.stream()
                .filter(f -> f.getFileName().equals(fileName))
                .findFirst()
                .orElseThrow(() -> new RuntimeException("File not found: " + fileName));
        return Files.readAllBytes(storageDir.resolve(meta.getFileStoredName()));
    } finally {
        lock.readLock().unlock();
    }
}

// uploadFile — wyłączny zapis per-plik + csvLock do metadanych:
public void uploadFile(FileMetadata meta) {
    ReentrantReadWriteLock lock = lockFor(meta.getFileName());
    lock.writeLock().lock();
    try {
        Files.write(storageDir.resolve(meta.getFileStoredName()), meta.getFileData());
        meta.setFileData(null);  // bajty zwolnione — brak wycieku pamięci
        files.add(meta);
        saveMetadataToCsv();     // wewnątrz: csvLock.lock() / unlock()
    } finally {
        lock.writeLock().unlock();
    }
}

Dlaczego lock nie jest usuwany z mapy po deleteFile(): inny wątek mógł już pobrać referencję do obiektu locka przed usunięciem go z mapy. Gdyby lock został usunięty, ten wątek działałby na "osieroconej" blokadzie i mógłby odczytać plik po jego fizycznym usunięciu. Zachowanie locka w mapie jest świadomym kompromisem: potencjalny drobny wyciek pamięci przy masowych operacjach w zamian za pełne bezpieczeństwo wątkowe.

2. Lista użytkowników — UserRepositoryImpl

Problem: Jednoczesna rejestracja i usuwanie użytkowników z wielu klientów może prowadzić do ConcurrentModificationException, a równoległy zapis CSV może nadpisać zmiany.

Rozwiązanie: CopyOnWriteArrayList dla listy użytkowników (bezpieczna iteracja) oraz synchronized na metodach save() i delete() — gwarantuje atomowość pary operacji: modyfikacja listy + zapis CSV.

3. Sesje użytkowników — AuthServiceImpl

Rozwiązanie: Dwie mapy ConcurrentHashMap (sesja→login, sesja→rola). Wszystkie operacje atomowe, bez potrzeby jawnej synchronizacji.

Brak zakleszczeń (Deadlock-free)

System nie jest podatny na zakleszczenia, ponieważ:

  • Hierarchia locków: wątek trzyma zawsze co najwyżej dwa locki jednocześnie — per-file lock (writeLock) i następnie csvLock. Kolejność akwizycji jest zawsze taka sama we wszystkich ścieżkach kodu, więc cykliczne oczekiwanie jest niemożliwe.
  • Blokady zawsze zwalniane w bloku finally — gwarantowane zwolnienie nawet przy wyjątku.
  • computeIfAbsent w ConcurrentHashMap jest atomowe — brak race condition przy tworzeniu nowego locka.

Zarządzanie pamięcią — kluczowe decyzje

  • Zwolnienie bajtów po uploadzie: Po zapisaniu pliku na dysk wywołujemy meta.setFileData(null). Bez tego każdy wgrany plik trzymałby bajty w heap przez cały czas działania serwera.
  • listFiles() korzysta z fileSize: Pole fileSize ustawiane przy tworzeniu obiektu lub ładowaniu z CSV, dzięki czemu mapToDTO() nie musi odczytywać fileData (które jest już null).
  • Bajty w RMI: Podczas uploadu/downloadu byte[] jest trzymany w pamięci JVM tylko przez czas trwania wywołania RMI — serializacja/deserializacja zarządzana przez JVM.
  • Limit 50 MB: Sprawdzany dwustronnie — klient sprawdza przed odczytem pliku z dysku (zapobiega ładowaniu gigantycznych plików do pamięci), serwer weryfikuje po otrzymaniu bajtów (ochrona przed obejściem walidacji klienta).

Ograniczenia

  • Brak TTL dla sesji — zalogowane sesje nie wygasają automatycznie.
  • Limit 50 MB per plik — bardzo duże pliki nie są obsługiwane.
  • Locki per-plik pozostają w mapie po usunięciu pliku — przy masowych operacjach upload+delete przez dłuższy czas mapa rośnie. Akceptowalne w środowisku laboratoryjnym; rozwiązaniem byłoby użycie WeakReference lub okresowe czyszczenie locków bez oczekujących wątków.
  • Hasła przechowywane w CSV plain-textem — brak hashowania.

8. Ograniczenia i testy poprawności

Maksymalna liczba jednoczesnych klientów

Java RMI tworzy wątek per wywołanie przy użyciu puli wątków JVM. Domyślnie brak twardego limitu — liczba jednoczesnych klientów ograniczona jest zasobami systemowymi. System działa stabilnie przy 10–20 jednoczesnych klientach w środowisku laboratoryjnym.

Test 1 — równoczesne pobieranie pliku (10 wątków)

Scenariusz: 10 wątków klienckich jednocześnie pobiera ten sam plik z serwera.

ExecutorService pool = Executors.newFixedThreadPool(10);
CountDownLatch latch = new CountDownLatch(1);
List<byte[]> results = new CopyOnWriteArrayList<>();

for (int i = 0; i < 10; i++) {
    pool.submit(() -> {
        try {
            latch.await(); // wszystkie wątki startują jednocześnie
            byte[] data = fileService.downloadFile("test.txt");
            results.add(data);
        } catch (Exception e) { e.printStackTrace(); }
    });
}
latch.countDown();
pool.shutdown();
pool.awaitTermination(30, TimeUnit.SECONDS);

assert results.size() == 10;
assert results.stream().allMatch(d -> d != null && d.length > 0);

Oczekiwany wynik: Wszystkie 10 wątków otrzymuje identyczne, kompletne bajty pliku. Brak wyjątków.

Dlaczego działa: Per-file readLock() pozwala wielu wątkom jednocześnie trzymać blokadę odczytu tego samego pliku — żaden nie blokuje pozostałych.

Test 2 — równoczesny pobieranie i usuwanie (TOCTOU)

Scenariusz: Wątek A pobiera plik, wątek B usuwa ten sam plik jednocześnie.

Bez ochrony (błędny kod): Możliwy scenariusz — A sprawdza findFileByName (plik istnieje), B wywołuje deleteFile (plik usunięty), A wywołuje downloadFile (RuntimeException — plik nie istnieje na dysku).

Z ochroną (findAndDownload + per-file lock): Operacja find+read odbywa się atomowo pod per-file read lock. deleteFile próbuje uzyskać write lock na ten sam plik — musi poczekać, aż A skończy pobieranie. Wynik: albo A pobierze plik w całości, albo B usunie go przed A i A dostanie czytelny błąd "File not found".

Test 3 — równoległe uploady różnych plików (per-file locking)

Scenariusz: 5 wątków wgrywa jednocześnie 5 różnych plików.

Zachowanie z globalnym lockiem (poprzednia wersja): Wszystkie 5 wątków kolejkuje się na jednym write locku — efektywnie sekwencyjne.

Zachowanie z per-file lockiem (aktualna wersja): Każdy wątek trzyma write lock tylko swojego pliku — wszystkie 5 operacji zapisu na dysk wykonują się równolegle. Jedyna synchronizacja to csvLock przy zapisie metadanych (trzymany przez ułamek sekundy).

ExecutorService pool = Executors.newFixedThreadPool(5);
CountDownLatch latch = new CountDownLatch(1);

for (int i = 0; i < 5; i++) {
    final String name = "file_" + i + ".txt";
    final byte[] data = ("content " + i).getBytes();
    pool.submit(() -> {
        try {
            latch.await();
            fileService.uploadFile(name, data); // równoległe — różne locki
        } catch (Exception e) { e.printStackTrace(); }
    });
}
latch.countDown();
pool.shutdown();
pool.awaitTermination(30, TimeUnit.SECONDS);

assert fileService.listFiles().size() == 5;

Oczekiwany wynik: Lista zawiera dokładnie 5 plików. CSV zawiera 5 wpisów. Brak wyjątków.

Test 4 — persystencja po restarcie

Scenariusz: Wgranie 3 plików i dodanie użytkownika, restart serwera, weryfikacja stanu.

  1. Uruchom serwer, zaloguj się jako admin.
  2. Wgraj 3 pliki, dodaj użytkownika test.
  3. Zatrzymaj serwer (Ctrl+C).
  4. Uruchom serwer ponownie.
  5. Zaloguj się — opcja 1. Lista plików powinna pokazać te same 3 pliki.
  6. Zaloguj się jako test — logowanie powinno się powieść.

Oczekiwany wynik: Stan sprzed restartu w pełni odtworzony. Komunikaty startowe: [Storage] Załadowano 3 plików z CSV. i [Storage] Załadowano 2 użytkowników z CSV.
